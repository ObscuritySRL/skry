# Ease-hunt #3 — drag-drop, events, capture/OCR, clipboard, cross-window — 2026-06-19

Third ease-hunt on the input/perception/process/multi-window surfaces. 5 candidates, zero overlap with hunts #1/#2.
4 SHIPPED (live-verified), 1 DECLINED on counter-evidence.

## Shipped (all live-verified, pushed)

1. **fix(mcp): copy_files {move}** — copy_files's description AND success message promised "copy/move" but
   writeClipboardFiles placed ONLY CF_HDROP, which Windows defaults to DROPEFFECT_COPY — so a "move X into
   Archive" task silently left the source behind. {move:true} now also stages a "Preferred DropEffect" =
   DROPEFFECT_MOVE (2) format (zero new FFI; RegisterClipboardFormatW/SetClipboardData already bound). Verified
   against the real clipboard: move=true → CF_HDROP + DropEffect=2; default → CF_HDROP only.
2. **feat(mcp): wait_for_process {gone}** — the lone wait tool with no negative edge. Waiting for an
   installer/build/conversion/launched-app to FINISH forced a hand-rolled poll over list_processes (400+ procs/
   tick), and wait_for_window {gone} silently LIES for a windowless process (resolves at once). waitForProcessGone
   opens a SYNCHRONIZE handle per match (OpenProcess) and polls the OS-signaled handle O(1) per tick. Verified:
   blocked 2695ms for a real `ping` exit vs waitForWindowGone's false-immediate 4ms.
3. **feat(mcp): ocr per-word click centres** — the ocr handler rendered only line.bounds; a multi-word line's box
   centre misses the wanted word by tens of px, forcing a second full OCR via click_text. The per-word boxes were
   already computed (OcrLine.words) and discarded. Now each multi-word line lists "word"@cx,cy. Verified live (VS
   Code: 95 multi-word lines, distinct centres) + a painted-window test.
4. **feat(mcp): attach owner breadcrumb** — "open a dialog, act, return to the parent" had no breadcrumb (every
   switch disposes the prior target; attach named only the dialog). Now an OWNED window's attach appends "(owned by
   0x<owner> "<title>" — attach {hWnd:0x<owner>} to return to the parent)" via a new ownerWindow() = direct
   GetWindow GW_OWNER. Non-owned windows get nothing (~0 tokens). Verified live on an owned popup pair.

## Declined — drag dwell (counter-evidence)

5. **Real-cursor drag dwell** — the candidate claimed dragTo fires LEFTUP with zero dwell so dwell-sensitive drops
   (reorder/Explorer/tree/taskbar) silently no-op, and pitched adding ~50ms dwells. DECLINED: the claim is
   UNVERIFIED — the panel only measured TIMING (~5ms), never a real drop that fails without dwell or lands with it.
   And there is COUNTER-EVIDENCE: example/drag-interpolated-stroke.integration.test.ts already proves a real drag
   LANDS without any dwell (it paints a continuous stroke in Paint). The "unconditional success message" concern is
   also weaker than framed — every input tool reports the input was sent and relies on the snapshot Δ for the
   effect. Adding speculative dwell to the proven input path (which could also perturb the existing drag tests'
   timing) is exactly the unproven change this session keeps cutting. REVISIT only with a concrete failing-drop
   repro on a real dwell-sensitive target (then add dwell + prove the drop now lands).
